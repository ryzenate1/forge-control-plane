## Architecture Changes

- Placement reservations are now durable control-plane resources.
- Scheduler-facing capacity is reservation-aware through `NodeCapacitySnapshot`.
- ReservationManager owns reservation lifecycle, metrics, and events.
- Cluster Manager creates a short-lived placement reservation around server creation.
- Migration Service creates target-node reservations and releases them on terminal state.
- Reservation events flow through the existing Event Bus and are persisted to the timeline by the Observability subscriber.

## Known Limitations

- Recovery Coordinator is not implemented.
- Failover Engine is not implemented.
- Auto Recovery is not implemented.
- Migration execution still does not move workloads.
- Reservation expiration runs inside the API process and has no leader election.
- Reservation metrics are process-local counters.
- Event Bus remains in-process.
- Restore ownership remains daemon-coupled and out of scope.
- Go files were not automatically formatted because `gofmt` is not installed in this environment.
- Build, test, lint, typecheck, docker, and validation commands were not run by instruction.

## Remaining Race Conditions

- Scheduler scoring and reservation creation are separate steps. Reservation creation is atomic and conflict-safe, but a stale scheduler decision may fail and require retry.
- Server creation reservations are short-lived and do not include a server ID until the server exists.
- Long-running future recovery/restore workflows will need reservation renewal.
- API-process-local expiration should be replaced by coordinated control-plane ownership before distributed API deployment.
