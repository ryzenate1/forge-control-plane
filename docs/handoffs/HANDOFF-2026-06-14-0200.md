## Important Findings

- GamePanel is not ready for Failover & Recovery execution.
- Heartbeat data is persisted, but there is no Heartbeat Expiry Engine.
- Current state can represent online/offline/degraded, but cannot reliably classify unreachable nodes.
- Placement reservations do not exist, so double placement and capacity races remain possible.
- Migration records and history exist, but migration execution, locks, placement commit, rollback, and restore are missing.
- Restore ownership is undefined and remains daemon-coupled.
- Timeline events are useful for visibility, but the in-process Event Bus is not reliable enough to drive failover execution.

## Blockers

- No heartbeat expiry evaluator.
- No node unreachable state or policy.
- No placement reservation table.
- No migration/reconciliation coordination lock.
- No recovery plan domain.
- No restore orchestration owner.
- No runtime-neutral archive/restore contract.

## Risks

- Automatic failover could trigger incorrectly without reliable heartbeat expiry.
- Concurrent placement or migration could overcommit nodes without reservations.
- Reconciler could act on servers that are involved in migration intent.
- Recovery execution could be split across handlers, Cluster Manager, Migration Service, Runtime Layer, and daemon APIs without a clear owner.

## Next Exact Step

Review `docs/FAILOVER_READINESS_AUDIT.md`. If approved, build the Heartbeat Expiry Engine next. Do not implement Failover & Recovery execution until heartbeat expiry, placement reservations, migration coordination, and restore ownership are reviewed.
