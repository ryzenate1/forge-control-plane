## Architecture Findings

- Placement ownership is split between Scheduler and Cluster Manager because allocation fallback remains in Cluster Manager and placement decisions are not durable.
- Node lifecycle ownership is mostly in Node Registry, but Scheduler and Node Registry duplicate placement eligibility logic.
- Reconciler owns desired-vs-actual comparison, but runtime actions still flow through Cluster Manager methods that also mutate actual state.
- Migration Service owns migration state and history, but migration execution is intentionally state-machine-only and does not move workloads.
- Runtime abstraction exists, but install, files, backups, realtime console/logs, some stats, and daemon heartbeat vocabulary remain daemon-coupled.
- Events and metrics are in-process and not durable.

## Blockers

- No durable placement reservations.
- No heartbeat-expiry/offline detector.
- No migration execution, backup transfer, restore orchestration, or server placement update.
- No migration/reconciliation lock or ownership marker.
- No durable event stream or operator-facing orchestration timeline.

## Risks

- Automatic failover could double-act without coordination between Reconciler, Cluster Manager, and Migration Service.
- Concurrent placement or migration could overcommit node capacity without reservations.
- Runtime expansion could spread provider-specific assumptions if remaining daemon coupling is not reduced.
- Ambiguous evacuation status semantics could confuse future automation.
