## Architecture Changes

- Heartbeat State is now a first-class persisted node signal independent from Node Actual State.
- Node Registry records heartbeat facts and no longer owns heartbeat-derived actual-state classification.
- Heartbeat Monitor owns node failure classification and compatibility mapping.
- Heartbeat transitions flow into the existing Event Bus and Observability timeline.
- Scheduler-facing compatibility is preserved through `online`, `degraded`, and `offline` actual states.

## Known Limitations

- Placement reservations are not implemented.
- Recovery Coordinator is not implemented.
- Failover Engine is not implemented.
- Migration execution still does not move workloads.
- Runtime expansion is not implemented.
- Heartbeat monitor metrics are process-local counters.
- Event Bus remains in-process.
- `gofmt` is not installed in this environment, so Go files were not automatically formatted.
- Build, test, lint, typecheck, docker, and validation commands were not run by instruction.

## Risks

- Without placement reservations, future recovery planning can still race with server creation or migration target selection.
- Heartbeat thresholds are process configuration and not yet persisted per cluster or node.
- Periodic heartbeat evaluation runs inside the API process and has no leader election.
- Existing daemon heartbeats remain the only recovery signal; no independent network probe exists.

Review Phase 11. If approved, implement Placement Reservations as the next foundation before Recovery Coordinator or Failover Engine.
