## Architecture Changes

- Cluster Manager owns server creation placement.
- Scheduler is now a real region-aware placement service.
- Capacity is queryable through Cluster Manager.
- Admin API exposes node and region capacity.
- Server responses expose a desired-state foundation without adding schema.

## Known Limitations

- No migrations were added, per instruction.
- Desired state is computed from existing `servers.status`, not stored in a dedicated column.
- Capacity is computed from current tables and treats unknown capacity as unconstrained for backwards compatibility.
- Allocation reservation still happens through existing server creation transaction.
- No reconciliation loop exists.
- No failover, event bus, runtime abstraction upgrade, predictive scheduling, or AI feature was started.
- `gofmt` is unavailable in this environment.

## Recommended Phase 3

Scheduler hardening:

- Make placement decisions more explainable.
- Add placement dry-run endpoint.
- Add richer capacity policy.
- Keep failover, event bus, runtime abstraction, reconciliation loops, and AI out of scope until explicitly requested.
