## Reconciliation Design

- Handlers request power changes through Cluster Manager.
- Cluster Manager remains the owner of infrastructure execution.
- Reconciler periodically refreshes nodes, capacity snapshots, and servers.
- Cluster Manager probes daemon stats as the available daemon-side status signal where applicable.
- Reconciler compares desired state and actual state.
- Reconciler performs only simple corrective server actions through Cluster Manager.
- Scheduler V1 continues to prevent placement on maintenance or draining nodes.
- No failover, migration, event bus, runtime abstraction, or AI behavior was added.

## Important Findings

- The current database schema has `servers.status`, `nodes.status`, `nodes.maintenance_mode`, and `nodes.draining`, but does not have dedicated desired-state and actual-state columns.
- Phase 3 explicitly prohibited migrations, so desired/actual state is projected from existing columns for compatibility.
- A future migration is required for true separately persisted desired and actual state.

## Blockers

- `gofmt` is not available in the current environment.
- Build, test, lint, typecheck, docker, and validation commands were intentionally not run.

## Risks

- Desired and actual server state cannot diverge durably until schema support exists.
- Node actual state can be approximated only from existing node status because maintenance/draining currently overwrite visible node status.
- The reconciliation loop currently has no leader election, so multiple API replicas would each run reconciliation if deployed together.
