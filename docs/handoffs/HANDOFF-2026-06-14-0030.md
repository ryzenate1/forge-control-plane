## Architecture Changes

- Runtime is now an API-side provider interface.
- Docker is now an adapter, not the orchestration boundary.
- Cluster Manager uses Runtime methods for power/delete/stats operations.
- Reconciler remains decoupled by calling Cluster Manager.
- Runtime Registry centralizes provider registration, lookup, capability checks, events, and process-local metrics.

## Important Findings

- Runtime coupling is reduced for Cluster Manager and Reconciler-facing operations.
- Handler-level daemon coupling remains for install, files, backups, logs, stats endpoint, and realtime proxy.
- Daemon internals remain Docker-backed.
- CPU resource semantics still use Docker-style `cpu_shares`.
- Runtime metrics and events are in-process only.

## Blockers

- `gofmt` is not available in this environment, so Go formatting could not be automated.
- Build, test, lint, typecheck, docker, and validation commands were intentionally not run.

## Risks

- Migration Engine should not begin until runtime-neutral archive/restore or snapshot contracts are designed.
- Failover should not begin until migration execution, heartbeat expiry, and leader election are designed.
- Existing handler-level daemon calls can still pull Docker assumptions back into new orchestration if reused directly.
